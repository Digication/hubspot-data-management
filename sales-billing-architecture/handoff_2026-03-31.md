## What was done today

### Issue #2 (High) -- Applied
The "Overdue for 90+ days" threshold in Section 2.4.2 was undefined. Now added:

- **Overdue status definition:** `paid_status` is set to Overdue automatically when invoice due date passes without payment. Staff can also set/clear manually.
- **Billing grace mode:** A per-deal boolean flag (`billing_grace_mode`, default off) set by staff in Campus admin. When enabled: the deal stays active for tier computation regardless of Overdue duration, no access restrictions, no user-facing messages. Intended for high-trust customers with known late payment patterns (e.g., BU paying 60 days after invoice due).
- **Active deal definition:** Explicitly defined -- not expired AND (not 90+ days Overdue OR billing_grace_mode = true).
- Added `billing_grace_mode` to Deal Properties (Section 2.6) and Appendix A.

**Key decision:** `billing_grace_mode` lives at the **deal level**, not company level. This was a deliberate choice by Jeff for granular control.

## Fact-Check Findings (verified against codebase at digication-monorepo)

A fact-check was run against the actual Campus codebase. Three findings change the picture:

### Finding 1: Issue #9 is more serious than the review suggests
**Claim:** Multiple Campus systems can map to one HubSpot Company.
**Reality:** The sync code at `HubspotSync.ts:130-140` actively **excludes** any systems that share a `salesCrmId` (HubSpot Company ID). It's treated as an error condition -- all systems in that situation are dropped from sync entirely. The spec assumes multi-system companies work; the code rejects them.

### Finding 2: Issue #7 is confirmed by code
**Claim:** Deactivation event might never reach HubSpot.
**Reality:** Confirmed. There is no deactivation-triggered HubSpot push anywhere in `campus-api/src`. No "final sync" pattern exists. When a system's `salesStatus` becomes `DEACTIVATED_*`, it silently drops from all future sync runs. HubSpot retains the last pre-deactivation snapshot permanently.

### Finding 3: Deal flow is read-only (correct), but companies are not
**Claim:** Campus never creates/modifies data in HubSpot.
**Reality:** Deals are read-only (confirmed). But Campus **does** create and update Companies in HubSpot via `batchApi.create()` and `imports.coreApi.create()`. The spec's description of deal flow is accurate; the broader "read-only" framing is imprecise.

### Issue #3 (Medium) -- Validation limited to 88 enterprise customers
**Direction from Jeff:** Understood as a gap (not intentional). The 88 is only enterprise_current records. The other ~1,500 self-signup/prospect records also need validation before Phase 3.
- Extend Phase 2 validation to all segments.
- Ready to apply.

### Issue #5 (Medium) -- Contact-level spend attribution
**Direction from Jeff:** Not actually a problem IF every deal is associated with a contact in HubSpot (which is the plan). In HubSpot, deals can be associated with both contacts and companies.
- Just need to document this as a dependency: spend computation requires deal-contact associations are always created.
- Ready to apply.

### Issue #7 (Medium) -- Stale data on deactivation
**Direction from Jeff:** The past reporting problem was about `sales_status` never being updated when customers churned, not about usage data going stale. The new architecture fixes this conceptually (system_state = Deactivated would be explicit). BUT the fact-check confirmed the risk: current sync excludes deactivated systems, so the deactivation state change might never reach HubSpot.
- Add requirement: Campus does one final sync push when a system transitions to Deactivated, before excluding it from future syncs.
- Ready to apply.

### Issue #9 (Medium) -- Multi-system aggregation
**Direction from Jeff:** Acknowledged as a bigger problem than initially scoped (especially given the fact-check finding). Jeff's initial thoughts on options:
1. Sum all deals together across systems
2. Designate which deals count and which don't
3. Other alternatives welcome

**Suggested approach (not yet confirmed by Jeff):**
| Field type | Strategy |
|---|---|
| Numeric (users, storage) | Sum across all production systems |
| Date fields | Earliest start, latest end |
| Status/categorical | Most significant value |
| Non-aggregatable (plan name) | Primary system only |

- Needs further discussion before applying.

### Issue #11 (Medium) -- Contract/deal data migration from HubSpot
**Direction from Jeff:** Confirmed needed. Campus must become the source of truth.
- Add Phase 1 step: audit existing HubSpot contract/deal data, plan one-time import into Campus.
- Owner: Sam + dev team.
- Ready to apply.
