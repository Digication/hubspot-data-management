# Review: Sales & Billing Architecture Spec v2

**Reviewer: Jeff Yan (with Claude) | March 26, 2026**
**Document reviewed:** Digication_Sales_Billing_Architecture_Spec_v2.docx (Google Drive)
**Purpose:** Evaluate whether v2 adequately addresses the 9 issues raised in Ly's engineering review of v1 (2026-03-25)

---

## Summary

v2 is a significant improvement over v1. All 9 engineering review issues have been addressed in the spec. The document is now architecturally sound and ready for developer review. A few items require developer input before implementation can begin (see Developer Action Items below).

---

## Issue-by-Issue Assessment

### Issue 1: Campus-side migration steps incomplete (Medium)
**Status: Fully resolved**

The migration plan (Section 3.2) now splits every phase into Campus-side and HubSpot-side steps. Phase 1 includes creating `system_type` in Campus and populating it from existing data. Phase 4 includes stopping writes to `sales_status` on the Campus side. No further action needed.

---

### Issue 2: Deal flow reversal not scoped as major build (High)
**Status: Resolved in spec**

New Section 2.5 explicitly calls out the deal flow reversal (Campus > HubSpot) as a major new integration. It documents:
- Current state (HubSpot > Campus, read-only)
- Target state (self-signup deals push from Campus to HubSpot via API)
- Coexistence model (enterprise deals stay manual in HubSpot during transition)
- Technical requirements (HubSpot API, associations, idempotency, error handling)

Open item #7 correctly assigns scoping/estimation to the dev team. No timeline concerns at this stage -- the focus is on getting the spec right.

---

### Issue 3: Where auto-computation runs is undefined (High)
**Status: Thoroughly resolved**

New Section 2.4 is the strongest addition in v2. It specifies:
- All computation runs in Campus (testable, version-controlled, debuggable)
- Two sync modes: immediate webhooks for self-signup events; periodic batch for enterprise (transitioning to webhooks over time)
- Detailed computation rules for Customer Tier (6-step priority logic), Lifecycle Stage, and Trial Status
- Manual override points with specific scenarios, actions, and effects
- Audit logging for all manual overrides

No further action needed. This section is implementation-ready.

---

### Issue 4: `system_type` field doesn't exist yet (Low)
**Status: Fully resolved**

Phase 1 now includes creating `system_type` (production/sandbox/internal) in the Campus database and populating it from existing `sales_status` values. Sandbox/internal values map accordingly; all others default to production.

---

### Issue 5: Trial fields partially exist in Campus (Low)
**Status: Resolved**

Phase 1 notes that `trial_start_date` and `trial_end_date` already exist. Only the three genuinely new fields (`trial_active`, `trial_status`, `trial_extension_count`) need to be created. This prevents duplicate field creation.

---

### Issue 6: QuickBooks integration is greenfield work (Medium)
**Status: Resolved via alternative approach**

Instead of building a full coded QuickBooks API integration, v2 proposes using Cowork agent-assisted workflow. The agent handles deal grouping, property extraction, QB invoice creation, and ID write-back.

This is the right approach for current volume (~100 enterprise invoices per year). These invoices are for enterprise plans only, not automatic credit card processing. Self-signup payments are handled through Stripe directly. If volume grows significantly in the future, a direct coded integration can be built then.

---

### Issue 7: WIP billing system not mapped to deal types (Medium)
**Status: Addressed with developer placeholder approach**

New Section 1.4 maps each spec deal type to the billing system with columns for Billing System Entity, What Exists, and What Needs Wiring. Developer placeholders (`[Developer: entity name]`, `[Developer: describe current state]`) create an actionable template the dev team must fill in. See Developer Action Items below.

---

### Issue 8: HubSpot sync depends on `sales_status` (High)
**Status: Well resolved**

Phase 2 now explicitly includes migrating sync filter logic from `sales_status`-based filtering to `system_type` + `system_state` filtering, with a clear note that this must be done before `sales_status` can be archived. Phase 4 adds a verification step to confirm all sync paths have been migrated.

---

### Issue 9: Label/value naming mismatch (Low)
**Status: Resolved**

The status table in Section 1.1 now has separate columns for Internal Value and Display Label, eliminating the ambiguity between `internal` ("Sandbox (Internal)") and `sandbox` ("Sandbox (Customer)").

## Developer Action Items

The following items require developer input before implementation begins. These are not spec gaps -- the spec is architecturally complete -- but they need developer knowledge of the codebase to finalize.

### Action 1: Fill in billing system entity mapping (Section 1.4)
**Owner:** Dev team
**What:** Section 1.4 contains a table mapping spec deal types to billing system entities. The columns `[Developer: entity name]` and `[Developer: describe current state]` need to be filled in with actual model/entity names from the Campus billing codebase.
**Why:** The integration work in Phase 2 depends on knowing exactly which billing system entities map to which HubSpot deal types.
**Blocks:** Phase 2 Campus-side work

### Action 2: Scope the deal flow reversal integration (Section 2.5)
**Owner:** Dev team
**What:** Section 2.5 describes a new Campus > HubSpot deal push integration. This is the largest new build item in the spec. The dev team should review the technical requirements listed in Section 2.5.3 and provide a scope estimate.
**Why:** This integration doesn't exist today. It requires HubSpot API integration for deal creation, association linking, error handling with retry logic, and idempotency guarantees.
**Blocks:** Phase 2 Campus-side work

### Action 3: Review auto-computation rules (Section 2.4)
**Owner:** Dev team
**What:** Section 2.4 contains detailed computation rules for Customer Tier (6-step priority logic), Lifecycle Stage (4-step logic), and Trial Status (5-step logic). Developers should review these rules against the current Campus data model to confirm they are implementable as written.
**Why:** These rules will be coded into Campus. Any edge cases or data model mismatches are cheaper to catch now than during implementation.

### Action 4: Validate open items #4 and #5 with Sam
**Owner:** Sam
**What:** Open item #4 (QuickBooks income categories) and #5 (contract field reconciliation) both require Sam's input. QB categories need validation against the actual chart of accounts. Existing HubSpot contract fields need to be mapped against the new Campus contract model.
**Why:** These are Phase 1 prerequisites that cannot be completed without finance/operations input.
