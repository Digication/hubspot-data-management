# Review: Sales & Billing Architecture Spec

**Reviewer Notes for Jeff Yan | March 25, 2026**

---

## Overall Assessment

This is a strong spec. The core diagnosis is right -- the current `sales_status` field tries to do too many jobs at once, and splitting it into separate concerns (customer tier, lifecycle stage, system state) is the correct architectural direction. The document is well-structured, the migration mapping is thorough, and the design principles are sound.

That said, after reviewing the current systems in detail, there are several areas where the spec would benefit from clarification or adjustment before implementation begins. None of these are deal-breakers, but addressing them now will avoid surprises mid-build.

---

## What the Spec Gets Right

1. **The problem diagnosis is spot-on.** The 19-value status field genuinely mixes unrelated concepts together. Separating tier, lifecycle, and system state into independent properties will make reporting reliable and reduce manual work.

2. **Contract management is a real gap.** There is no structured contract tracking in either Campus or HubSpot today. Making contracts a first-class entity is overdue and the proposed data model is solid.

3. **Contact sync is a real gap.** There is currently no infrastructure for syncing contact data to HubSpot. The two-field approach (Sales Role vs. Operational Role) is a clean design that keeps sales workflows separate from operational data.

4. **"Campus as source of truth" is the right call.** This aligns with how the systems already work and avoids the mess of having two systems fight over who owns the data.

5. **The Self-Signup Pipeline is needed.** Forcing self-service customers through enterprise sales stages doesn't make sense. A dedicated pipeline with automated, light-touch stages is the right approach.

---

## Areas That Need Attention

### Issue 1: The spec describes `sales_status` as a HubSpot field, but it originates in Campus

> *Section 1.1: "The current sales_status field on HubSpot Company records contains 19 possible values."*

In practice, this field lives in the Campus database first and gets synced to HubSpot. The migration plan (Section 3.2) focuses heavily on HubSpot-side changes -- creating new properties, running migration scripts, archiving the old field. But the Campus side also requires changes: the database column, the default values assigned to new systems, and the logic that uses this field to determine which systems are eligible for syncing.

**Recommendation:** Add Campus-side migration steps alongside the HubSpot-side steps in each phase.

---

### Issue 2: The deal data flow is being reversed, and this isn't flagged as a major change

> *Section 2.4: "When a customer makes a purchase...each distinct product/service becomes its own deal in HubSpot."*
>
> *Section 4: "Deals are created in HubSpot (either automatically from self-signup/Stripe or manually for enterprise sales)."*

Today, deals only flow in one direction: from HubSpot down to Campus (read-only). The sales team creates deals in HubSpot, and Campus downloads them for reference.

The spec proposes the opposite direction for self-signup: billing events in Campus would create deals and push them up to HubSpot. This is a fundamental change to the integration architecture -- it requires building an entirely new sync capability that doesn't exist today.

**Recommendation:** Call this out explicitly as a major new integration to build. It should be scoped and estimated separately, not treated as a configuration change. It may also need to coexist with the current manual deal creation workflow for enterprise sales.

---

### Issue 3: "Auto-computed" properties need a clear home

> *Section 2.3: "Customer Tier (auto-computed)... Lifecycle Stage (auto-computed)"*
>
> *Section 3.2, Phase 2: "Build auto-computation workflow in HubSpot"*

The spec says these properties are auto-computed but is inconsistent about where that computation happens. One line suggests HubSpot workflows. But the current sync between Campus and HubSpot runs on scheduled batches (daily, weekly, etc.), not in real-time.

This matters because:

- If computation runs as **HubSpot workflows**, the logic lives outside your codebase -- harder to test, version, and debug.
- If computation runs in **Campus and syncs on the current schedule**, there could be hours of delay between a customer purchasing and their status updating.
- If you need **real-time updates**, that requires new webhook-based infrastructure.

**Recommendation:** Make a clear decision on where auto-computation lives and document it. This affects the architecture of Phases 2 through 4.

---

### Issue 4: The `system_type` field is referenced but doesn't exist yet

> *Section 2.2: "A new Campus field 'system_type' (production/sandbox/internal) determines sync eligibility."*

Today, the only way to tell if a system is a sandbox or internal is through the `sales_status` field itself. The spec correctly proposes a new dedicated field, but the migration plan doesn't include the step of creating this field and populating it from existing data.

**Recommendation:** Add `system_type` creation and data population to Phase 1 or Phase 2 of the migration.

---

### Issue 5: Trial tracking partially exists in Campus already

> *Section 2.3, Property 2b: Proposes five trial properties (trial_active, trial_status, trial_start_date, trial_end_date, trial_extension_count)*

Campus already tracks trial start and end dates at the system level. Two of the five proposed fields already exist. The remaining three (trial_active, trial_status, trial_extension_count) are genuinely new.

**Recommendation:** Note which fields already exist so the implementation team builds on what's there rather than creating duplicates.

---

### Issue 6: QuickBooks integration has no existing foundation

> *Section 4: Describes a deal-to-QuickBooks invoice automation with income category mapping.*

There is currently zero QuickBooks integration in the codebase -- no API connection, no invoice sync, nothing. The spec presents this as a mapping and workflow exercise, but building a reliable connection to QuickBooks (authentication, API integration, invoice creation, error handling, reconciliation) is a substantial project.

The 8-week migration plan in Section 3.2 doesn't include any QuickBooks work, yet Section 4 describes a full integration.

**Recommendation:** Either add a dedicated phase for QuickBooks or explicitly note that it's out of scope for the initial 8-week migration and will follow as a separate project.

---

### Issue 7: The spec should reference what the WIP billing system already handles

> *Section 1.3: "The new billing system creates systems and processes payments automatically"*
>
> *Section 3.2, Phase 2: "Update the new billing system to create deals with the new properties"*

The spec mentions "the new billing system" in a few places but doesn't describe what it already covers. The billing system under development already includes plan management, license allocation, AI credit tracking with expiry, Stripe subscriptions, and invoice records. A brief mapping showing how these existing billing concepts relate to the proposed HubSpot deal types would help the implementation team understand what needs to be wired up vs. built from scratch.

**Recommendation:** Add a short table mapping the billing system's existing entities to the spec's deal types (Enterprise License, Self-Signup License, AI Credits, One-Time Fee).

---

### Issue 8: The HubSpot sync itself depends on `sales_status` -- you can't just archive it

> *Section 3.2, Phase 4: "Stop setting the legacy sales_status field on new records" and "Mark the legacy sales_status field as 'archived' in HubSpot"*

The current HubSpot sync uses `sales_status` to determine which systems are eligible to sync at all. Deactivated and sandbox systems are excluded based on their status value. If you archive this field without first replacing the sync filter logic, the sync will break or start syncing everything unfiltered.

**Recommendation:** Before Phase 4 cutover, the sync filtering logic needs to be migrated to use the new `system_type` and `system_state` fields instead. This should be part of Phase 2 or 3.

---

### Issue 9: Minor -- Status labels in the spec don't exactly match the internal values

> *Section 1.1: Lists "internal" as "Sandbox (Internal)" and "sandbox" as "Sandbox (Customer)"*

The labeling in the spec table could cause confusion during migration, since both sandbox-related values use the word "Sandbox" as their display label, but they map to different internal values. Worth a quick clarification pass to ensure the migration team reads them correctly.

## Priority Summary

| Priority | Issue | Risk |
|----------|-------|------|
| **High** | Deal flow reversal not scoped as a major build (Issue 2) | Could derail the timeline if discovered mid-implementation |
| **High** | HubSpot sync filtering depends on `sales_status` (Issue 8) | Blocks the Phase 4 cutover |
| **High** | Where auto-computation runs is undefined (Issue 3) | Affects architecture decisions across multiple phases |
| **Medium** | Campus-side migration steps are incomplete (Issue 1) | Implementation team may miss required changes |
| **Medium** | WIP billing system not mapped to deal types (Issue 7) | Developers won't know what to connect |
| **Medium** | QuickBooks is greenfield work, not in the 8-week timeline (Issue 6) | Scope creep risk |
| **Low** | `system_type` field needs creation + data migration (Issue 4) | Missing migration step |
| **Low** | Trial fields partially exist already (Issue 5) | Minor duplicate work risk |
| **Low** | Label/value naming mismatch (Issue 9) | Potential confusion during migration |

---

## Bottom Line

The spec provides a solid target architecture. The main risks are not in the *what* but in the *how* -- specifically, the deal flow reversal (Issue 2), the sync dependency on the field being retired (Issue 8), and the undefined location of auto-computation (Issue 3). Addressing these three items before implementation starts would significantly reduce the chance of mid-project surprises.
